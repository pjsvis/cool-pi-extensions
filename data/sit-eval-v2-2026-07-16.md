## Commentary

### The contradiction blind spot is resolved

The v1 eval showed SIT-002 (contradiction) at 62% — the weakest ingestion test. The v2 eval tells a different story. With the right system prompt priming (the v2 fixture's `system_prompt_append` explicitly mentions "contradictory constraints that cannot be simultaneously satisfied"), all four contradiction tests pass at 94–100%:

- SIT-002 (dependency vs feature): 100%
- SIT-007 (scale): 100%
- SIT-008 (temporal): 100%
- SIT-009 (resource): 94%

The contradiction blind spot was a **priming gap**, not a capability gap. When the protocol explicitly mentions contradiction detection, the models do it. This is the finding that should inform the next protocol edit — not by adding a new capability, but by making the existing capability explicit.

### The delivery gate is strong

The delivery tests show 86–100% pass rates. The decisiveness tests (SIT-010, SIT-011, SIT-013) — which were specifically designed to catch the "both-sides decoration" failure mode — pass at 94–100%. The Impartial Spectator, when properly primed, takes the position the evidence supports.

### DOT tests: fidelity perfect, ambiguity is the weak point

**SIT-014 (DOT fidelity): 17/17 (100%).** Every model that produced a valid response generated parseable, accurate DOT code. The `dot_parse` assertion (which runs the output through graphviz) passed every time. This is the grader-independent verification path working as designed — no grader needed, just `dot -Tsvg`.

**SIT-015 (DOT ambiguity): 13/17 (76%).** This is the weakest result in the entire eval. When given an ambiguous system description ("a service that talks to a database and a cache, there's also a queue involved somehow"), 4 models fabricated the relationships instead of asking for clarification or representing the ambiguity honestly. This is the same failure mode as EDI-005 (provenance fabrication) but in a formal-artifact context — the model would rather produce a complete-looking diagram than admit it doesn't know how the queue connects.

This is the genuine remaining weakness: models default to producing complete artifacts even when the input is incomplete. The ingestion gate catches this in prose ("what is frobnitz?"), but the delivery gate doesn't catch it in DOT — the model produces a diagram that looks complete but misrepresents the territory.

### The provider-failed models

4 models (glm-4.7, glm-5, glm-5.1, kimi-k2.6) have partial data due to persistent OpenRouter provider errors. These are infrastructure failures, not model failures. When provider stability improves, re-run `bash scripts/sit-eval-resumable.sh` and it will complete them. The resumable script now correctly skips models with 15/15 valid responses (responseLength > 0).

## What this tells us about the Edinburgh Protocol

The agreement holds. The ingestion gate (no shite in) is at 86–100% across all test flavors. The delivery gate (good stuff out) is at 86–100% for all tests except DOT ambiguity. The protocol works.

The one actionable finding: **DOT ambiguity (SIT-015) reveals a delivery-gate weakness** that the protocol should address. The model produces a complete-looking Thing from incomplete input rather than representing the incompleteness. This is "decorated Stuff" in its most literal form — a diagram that looks like a Thing but is fabricated.

The contradiction blind spot from v1 was a priming issue, not a capability issue. This suggests the protocol's next edit should make existing capabilities explicit rather than adding new ones.
