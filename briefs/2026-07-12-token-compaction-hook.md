# Brief: Scoped Token Compaction & Multi-Agent Memory Architectures

## 1. Objective
Design a low-frequency, batch-driven token compaction mechanism that leverages free infrastructure (e.g., NVIDIA Build endpoints) to prune conversational noise ("barnacles") while utilizing branch-native isolation to prevent repository clutter on the `main` branch.

---

## 2. Context & The Human Steering Substrate
In high-frequency conversational development, the human carries the primary semantic workload. This exploration leaves a trail of high-entropy noise—incorrect assumptions, deprecated code paths, and diagnostic verbosity. 

By passing the raw execution log through a cheap commodity reasoning model at task boundaries (e.g., `td handoff`), we extract a lossy, high-signal summary of state. This summary replaces raw chat logs, clarifying context history for the model without leaking token waste into subsequent turns.

---

## 3. The Branch-Isolated Memory Sidecar

To eliminate multi-agent write contention and prevent context contamination across parallel tasks, all intra-task memory is bound to the unique task IDs generated by `marcus/td`.

```text
[ Feature Branch: task/101-db ] ───► Writes to: .td-memory/101.md ───► Pushed to Origin (Failover Safe)
                                                                                  │
                                                                       (Squash Merge Boundary)
                                                                                  ▼
[ Production Branch: main ]     ◄─────────────────────────────────────── [ Clean Commit Message ]
                                                                       ( .td-memory/ is dropped )

```

### The Storage Lifecycle

* **Location:** Active intra-task memory is written to `.td-memory/${TASK_ID}.md`.
* **Version Control:** This folder is tracked by Git *only* within the lifecycle of the active feature branch.
* **Portability:** Committing and pushing this file to the remote feature branch ensures that if an environment fails or a handoff occurs, the incoming agent can run `td current` and instantly pull down the clean, de-barnacled summary history.

---

## 4. The Merge Gate & The Review Payoff

The `.td-memory/` file serves a critical purpose during the Peer or Agent Review phase before being discarded.

### 1. Eliminating Review Replays

When a review agent assesses a Pull Request, it typically only sees the final diff. By feeding the review agent the compacted `.td-memory/task-XYZ.md` sidecar alongside the diff, you provide a map of **rejected hypotheses**. The reviewer instantly sees what was already tried and discarded, focusing its attention entirely on validating the chosen path.

### 2. Compressing to the Essentials

When the task is approved, the repository utilizes a **Squash-and-Merge** execution step.

* The individual "fix typo" and "compaction pass" commits are melded into a single clean production commit.
* The explicit markdown text within the sidecar file is used to auto-populate a comprehensive, structured PR description.
* **The Delete Phase:** The feature branch is deleted upon merging. The `.td-memory/` folder never enters the active file tree of the `main` branch.

---

## 5. Architectural Metrics

| State | Scope | Target Action |
| --- | --- | --- |
| **Branch Active** | `.td-memory/task-XYZ.md` exists. | **Isolate.** Feed to active agent tokens to bound the context window. |
| **PR Review** | Diff + Sidecar context. | **Audit.** Review agent verifies implementation against rejected alternatives. |
| **Merge Complete** | Branch deleted. | **Vaporize.** The scratchpad is cleared from disk; essentials are preserved in the squash commit message. |

# Appendix: Spatial Waypoint & Location Tag Injection Schema

## 1. Objective
Define the mechanical injection layer for introducing deterministic spatial anchors (`[LOC: ...]` and `[WAYPOINT: ...]`) into the active developer-agent dialog stream. This ensures the raw log contains reliable data bounds for low-frequency compaction passes.

## 2. Manual Shell Overrides (Human-Driven Anchors)
To allow the human to quickly mark a logical milestone or force a context shift without leaving the terminal, we map two lightweight shell functions into the local environment profile (`.zshrc` or `.bashrc`).

These functions query `marcus/td` to determine the active task context and append the structural token directly into the current session log stream.

```bash
# Append a spatial file coordinate to the active task log
loc() {
  local target_file="${1:-$(git diff --name-only | head -n 1)}"
  if [ -z "$target_file" ]; then
    echo "Aborted: No file specified and git tree is clean."
    return 1
  fi
  
  local task_id=$(td current)
  echo "\n[LOC: ${target_file}]" >> ".td-memory/${task_id}.raw.log"
  echo "System Anchor: Bound context to ${target_file}"
}

# Append a logical milestone/pivot marker to the active task log
wp() {
  if [ -z "$1" ]; then
    echo "Usage: wp \"Description of the milestone or pivot\""
    return 1
  fi
  
  local task_id=$(td current)
  echo "\n[WAYPOINT: ${1}]" >> ".td-memory/${task_id}.raw.log"
  echo "System Anchor: Dropped waypoint \"${1}\""
}
